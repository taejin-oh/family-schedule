# 안드로이드 앱 전환 타당성 분석 (오프라인 단독 동작)

> 목표: 현재 PC(맥)에서 돌아가는 은채 숙제 관리 프로그램을, **인터넷 연결 없이 개별 핸드폰에서 단독으로 돌아가는 안드로이드 앱**으로 만들 수 있는지, 만든다면 어떤 기능이 빠지는지 코드 전반을 분석한 문서.
>
> 분석 시점 기준 브랜치: `android-app` (main 기준)

---

## 0. 한 줄 요약

대부분의 화면·기능(숙제/학원/매일·매주 할 일/시간표/스티커 보상/대시보드)은 **전부 로컬 데이터 기반이라 오프라인 단독으로 충분히 가능**하다. 다만 **AI 숙제 자동 추출**, **텔레그램 알림**, **백그라운드 워커**, **외부 Agent API** 이 4가지는 PC/서버/인터넷에 묶여 있어 그대로는 폰에서 못 돌고, 각각 대체 방식이 필요하다.

가장 현실적인 결론:
- **이미 PWA다.** `public/manifest.webmanifest`(`display: standalone`) + `public/sw.js`(서비스워커 오프라인 캐시) + `app/layout.tsx`의 메타데이터가 다 갖춰져 있다. 즉 "안드로이드에 설치되는 앱"이라는 껍데기는 이미 존재한다.
- 진짜 문제는 **데이터 계층과 서버 로직이 Node 전용(better-sqlite3, sharp, child_process)** 이라 폰 안에서 못 돈다는 것. 폰 단독 앱이 되려면 이 백엔드를 폰에서 돌 수 있는 형태로 다시 구현해야 한다.

## 1. 현재 아키텍처와 "PC 종속성"의 정체

### 1.1 스택
- **프론트:** Next.js 16 (App Router, RSC, Server Actions) + React 19 + Tailwind v4 + shadcn-ui. 한국어 UI, Asia/Seoul 고정.
- **데이터:** SQLite 2개
  - `data/app.db` — 도메인 데이터 (`server/db/schema.ts`)
  - `data/jobs.db` — 잡 큐 + 다이제스트 중복방지 로그 (`server/jobs/schema.ts`)
  - 드라이버: `better-sqlite3` (**네이티브 C++ 모듈, Node 전용**)
- **백그라운드 워커:** `instrumentation.ts` → `server/worker/run.ts`. Next 프로세스 안에서 1초 폴링 루프로 상시 동작.
- **AI 추출:** `server/llm/claude-cli.ts` — 로컬 `claude -p` CLI를 `child_process.spawn`으로 호출.
- **알림:** `server/notifications/*` — `https://api.telegram.org`로 fetch.
- **사진:** `server/storage/photos.ts` — `sharp`(**네이티브 모듈**)로 리사이즈해 `data/photos/`에 파일로 저장.

### 1.2 왜 "PC에서만 제한적으로" 도는가
세 가지가 전부 **로컬 머신 자원**을 직접 만진다:

| 종속성 | 코드 위치 | 폰에서 안 되는 이유 |
|---|---|---|
| `child_process.spawn('claude', ...)` | `server/llm/claude-cli.ts:75` | 폰에 claude CLI도 없고, 임의 프로세스 실행도 안 됨 |
| `better-sqlite3` | `server/db/client.ts` | 네이티브 Node 애드온. 안드로이드 런타임에서 못 돎 |
| `sharp` 이미지 처리 | `server/storage/photos.ts` | 네이티브 libvips 바인딩. 모바일 빌드 불가 |
| `fetch(api.telegram.org)` | `server/notifications/telegram.ts` | 인터넷 필요 (오프라인 위배) |
| 상시 폴링 워커 | `server/worker/run.ts` | 폰 OS는 상시 백그라운드 프로세스를 허용 안 함 |

즉 UI는 모바일 친화적인데, **백엔드 전체가 "한 대의 Node 서버가 항상 켜져 있다"는 전제** 위에 서 있다. 이게 PC 종속성의 핵심이다.

## 2. 안드로이드 앱으로 만드는 3가지 경로

### 경로 A — PWA를 "오프라인 우선"으로 강화 (권장 출발점)
이미 PWA이므로 가장 적은 변경으로 시작 가능. 단, **현재는 진짜 오프라인이 아니다**:
- `sw.js`는 HTML/정적자산만 캐시하고 `/api/*`와 Server Action은 **일부러 가로채지 않는다**(주석에 명시). 즉 데이터 읽기/쓰기는 전부 켜져 있는 Next 서버로 간다.
- 데이터가 서버 SQLite에 있으므로, 서버(=PC)가 꺼지면 화면은 떠도 **데이터를 못 읽는다.**

→ 진짜 폰 단독이 되려면 **데이터를 폰 안(IndexedDB 등)으로 옮기고, Server Action 호출을 클라이언트 로컬 로직으로 대체**해야 한다. 이건 사실상 프론트의 데이터 계층 재작성이다.

### 경로 B — Capacitor/Tauri로 웹앱을 네이티브 셸에 포장
현재 React UI를 거의 그대로 쓰고, 네이티브 SQLite 플러그인(`@capacitor-community/sqlite`)·카메라·로컬 알림 API를 붙이는 방식. **UI 재사용률이 가장 높다.** 단, Server Action/RSC 의존을 클라이언트 사이드 데이터 접근으로 바꿔야 함 (경로 A와 같은 작업).

### 경로 C — 네이티브 재작성 (Kotlin/Compose 또는 React Native)
UI까지 다시 만든다. 비용이 가장 크지만 모바일 UX·백그라운드 작업·온디바이스 AI 통합이 가장 깔끔. 단독 1인용 앱에는 과투자.

> **추천:** 경로 B(Capacitor) 또는 강화된 경로 A. 둘 다 핵심 과제는 동일 — **"서버 SQLite + Server Action"을 "폰 로컬 DB + 로컬 로직"으로 옮기는 것.**

## 4. 오프라인 단독에서 "빠지거나 바뀌는" 기능 상세

### 4.1 AI 숙제 자동 추출 — 가장 큰 손실 ❌

- **현재:** 사진/PDF 업로드 → 잡 생성 → 워커가 `claude -p`를 5분 타임아웃으로 호출 → 학원 syllabus의 표 구조까지 읽어 숙제 항목(JSON)으로 추출 → 신뢰도/근거까지 매김(`server/llm/prompt.ts`, `claude-cli.ts`).
- **오프라인 단독에서 불가능한 이유:** claude CLI는 폰에 없고, 모델은 클라우드 호출이라 인터넷이 필요. 이게 이 프로그램의 가장 큰 부가가치인데 **오프라인 제약 시 직격탄**.
- **대체 옵션 (현실성 순):**
  1. **수동 입력 우선화** — AI 없이도 숙제 추적의 핵심은 굴러간다. 수동 입력 UX를 1급 시민으로 끌어올리는 게 가장 싸고 확실.
  2. **온디바이스 OCR** — Google ML Kit Text Recognition(온디바이스, 오프라인, 한국어 지원)으로 사진에서 텍스트만 뽑아 **반자동 입력 보조**(항목 분리·날짜 추측은 휴리스틱). 진짜 "이해"는 아니지만 타이핑 절감.
  3. **하이브리드(인터넷 있을 때만)** — 와이파이 잡히면 클라우드 LLM(Claude API 등)으로 추출, 없으면 수동. "완전 오프라인"은 깨지지만 실사용 만족도는 가장 높음.
  4. **온디바이스 LLM** — 현시점 폰에서 PDF 표 추론 품질은 부족. 비권장.

> 정리: **완전 오프라인을 고수하면 자동 추출은 사실상 포기하고 수동 입력(+OCR 보조)로 대체**. 자동 추출을 살리려면 "인터넷 있을 때만 AI"라는 타협이 필요.

### 4.2 알림 (텔레그램 다이제스트 + 학원 리마인더)

- **현재:** 워커가 아침/저녁 정해진 시각에 다이제스트를 만들어 텔레그램으로 전송하고, 학원 시작/종료 ±N분 알림도 텔레그램으로 보냄.
- **오프라인 단독에서:**
  - 텔레그램 전송 = 인터넷 필요 → **제거 또는 선택 기능화.**
  - 그러나 알림의 *목적*(아침 브리핑, 학원 알림)은 **안드로이드 로컬 알림(Notification + AlarmManager/WorkManager)으로 더 잘 구현된다.** 텔레그램 봇 셋업도 필요 없어지고 부모 폰에서 바로 뜸.
- **결론:** 텔레그램은 빠지지만, **로컬 푸시 알림으로 대체하면 기능적으론 오히려 개선.** 다이제스트 본문 생성 로직(`digests.ts`)은 순수 함수에 가까워 폰에서 재사용 가능.

### 4.3 백그라운드 워커 ❌→⚠️

- **현재 하는 일** (`server/worker/run.ts`): ① 추출 잡 처리 ② 다이제스트 전송 ③ 학원 리마인더 ④ 매일 04:00 정리(보관/사진삭제) ⑤ 죽은 잡 회수.
- **폰에서:** OS가 상시 폴링 프로세스를 허용하지 않음. 대신:
  - 추출 잡 → (AI 빠지면) 불필요, 또는 인터넷 있을 때 즉시 처리.
  - 다이제스트/리마인더 → **WorkManager 주기 작업 + 정확 알람**으로 스케줄.
  - 정리(아카이브/사진삭제) → **앱 실행 시점에 1회 실행**으로 충분(1인용이라 실시간성 불필요).
- **결론:** "상시 워커"라는 개념은 사라지지만, 폰의 정상적인 스케줄링 수단으로 전부 대체 가능.

### 4.4 외부 Agent API ❌ (제거 대상)
`/api/agent/**`는 OpenClaw 같은 외부 에이전트가 베어러 토큰으로 데이터를 읽고 쓰는 통합 지점. **폰 단독 앱에선 존재 이유가 없어 제거.** (반대로, 폰 앱을 "클라이언트"로 두고 PC를 동기화 서버로 쓰는 하이브리드라면 이 API가 동기화 채널의 토대가 될 수 있음 — §6.)

## 5. 데이터 계층 — 가장 큰 실제 작업량

진짜 공수는 화면이 아니라 **데이터 접근 방식 전환**에 있다.

- 현재: 화면(RSC/Server Action) → `getDb()`(better-sqlite3) → SQLite 파일. 전부 **서버에서** 실행.
- 폰 단독: 같은 스키마를 **폰 로컬 DB**(Capacitor SQLite, 또는 React Native이면 op-sqlite/expo-sqlite, 순수 PWA면 IndexedDB)에 두고, `server/actions/*.ts`의 쿼리 로직을 **클라이언트에서 도는 동등한 함수로 포팅**.
- 다행인 점:
  - 스키마가 작고 명확하다 (`server/db/schema.ts` 178줄, 10개 테이블).
  - 시간을 `'HH:MM'`/`'YYYY-MM-DD'` **문자열**로 저장하는 규칙이라 타임존 직렬화 이슈가 적다.
  - Drizzle ORM을 쓰므로, Drizzle이 도는 모바일 SQLite 드라이버를 고르면 쿼리 상당수를 재사용할 수 있다.
- 작업 성격: 비즈니스 로직(필터/버킷/중복제거/마감일 보정)은 대부분 순수 함수라 이식 가능. **"서버에서 돈다"는 가정만 걷어내면 됨.**

## 6. (대안) 완전 오프라인 대신 "폰 클라이언트 + PC 동기화"

사용자가 *완전* 오프라인을 절대조건으로 두지 않는다면, 가장 ROI 높은 그림은:
- 폰 앱은 **로컬 DB로 평소엔 오프라인 동작** (모든 조회·수동입력·완료체크 즉시 됨).
- 와이파이/집 네트워크가 잡히면 PC의 기존 Next 서버와 **동기화**(이미 있는 `/api/agent/**`를 동기화 API로 확장).
- AI 추출·텔레그램은 **PC가 켜져 있을 때만** 수행 → 무거운 건 PC가, 일상 사용은 폰이.

이러면 "빠지는 기능"이 거의 없고, 폰은 인터넷 없어도 일상 사용에 지장 없다. **완전 단독보다 현실적이고 손실이 적다.**

## 7. 권장 로드맵 (완전 오프라인 단독 기준)

1. **데이터 계층 포팅** — 스키마를 폰 로컬 SQLite로, Server Action 쿼리를 클라이언트 로직으로. (최대 공수, 모든 것의 전제)
2. **셸 선택 & 포장** — Capacitor로 현 React UI 재사용 + 네이티브 SQLite/카메라/로컬알림 플러그인.
3. **수동 입력 UX 승격** — AI 부재를 메우는 가장 중요한 보완. (선택) ML Kit OCR로 반자동 보조.
4. **로컬 알림 전환** — 텔레그램 다이제스트/학원 리마인더를 WorkManager+Notification으로.
5. **정리 작업 재배치** — 워커의 04:00 정리를 앱 시작 시 1회 실행으로.
6. **불필요 코드 제거** — `/api/agent/**`, 텔레그램, claude-cli, 상시 워커.
7. (선택) **하이브리드 AI** — 인터넷 있을 때만 클라우드 LLM 추출.

## 8. 결론

- **가능한가?** 예. UI·로직의 대부분은 오프라인 단독으로 충분히 동작하고, 앱 껍데기(PWA)도 이미 있다.
- **무엇이 빠지나?** 본질적으로 빠지는 건 **AI 자동 추출(완전 오프라인 시)** 와 **텔레그램 전송** 둘뿐. 학원 알림·다이제스트·백그라운드 정리는 *기능이 사라지는 게 아니라 폰 네이티브 수단으로 대체*되며 오히려 더 자연스러워진다. 외부 Agent API는 단독 앱엔 불필요.
- **가장 큰 일은?** 화면이 아니라 **서버 SQLite + Server Action을 폰 로컬 데이터 계층으로 옮기는 재구현**.
- **현실적 권고:** 완전 오프라인을 절대조건으로 두기보다, **폰 로컬 우선 + 인터넷 있을 때만 AI/동기화**(§6)가 손실 없이 가장 실용적.
